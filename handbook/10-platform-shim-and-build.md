# 10 — Platform Shims and Build

The orchestrator (Node) is small. Two platform shims carry the heavy lifting:

| Shim                            | Language | Bundle path                       | Purpose                                                                                                   |
| ------------------------------- | -------- | --------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `apply-seccomp-{x64,arm64}`     | C        | `vendor/seccomp/<arch>/`          | Linux seccomp filter installation + nested PID namespace setup                                           |
| `srt-win-{x64,arm64}.exe`       | Rust     | `vendor/srt-win/<arch>/`          | Windows: WFP install/verify, sandbox user provisioning, two-hop launch, ACL manipulation, CA-trust install |

The packages `vendor/seccomp-src/` and `vendor/srt-win-src/` are the unbuilt source.

## 10.1 `vendor/seccomp-src/apply-seccomp.c`

A single 600-LOC C file producing a **static, libc-independent** ELF binary.

### Build

```bash
# At vendored C source dir:
mkdir -p ../seccomp/x64 ../seccomp/arm64
gcc -static -O2 -D__x86_64__ -o ../seccomp/x64/apply-seccomp apply-seccomp.c
aarch64-linux-gnu-gcc -static -O2 -D__aarch64__ -o ../seccomp/arm64/apply-seccomp apply-seccomp.c
```

The npm package's `vendor/seccomp/build.ts` orchestrates this for both architectures when running on Linux. macOS/Windows CI does not invoke it (they don't ship the binary).

### BPF Filter Source

`seccomp-unix-block.c` is the BPF program emitted by `seccomp` itself. The compiled output is **baked into** the C binary at build time:

```c
// Generated by: gcc -c -o seccomp-unix-block.o seccomp-unix-block.c
// then: objcopy -O binary -j .data seccomp-unix-block.o unix-block-bpf.bin
// finally: xxd -i unix-block-bpf.bin > unix-block-bpf.h
#include "unix-block-bpf.h"   // ← contains `unsigned char unix_block_bpf[] = { ... }`
```

This is the right ABI: `unix_block_bpf` is exactly the byte sequence libseccomp produces, but as a `const` array in `.rodata` so we can pass it to `prctl(PR_SET_SECCOMP, BPF, &prog)` directly. No file IO at runtime.

### Architecture-specific macros

```c
#if defined(__x86_64__)
#  define SRT_AUDIT_ARCH AUDIT_ARCH_X86_64
#  define SRT_HAS_X32 1
#elif defined(__aarch64__)
#  define SRT_AUDIT_ARCH AUDIT_ARCH_AARCH64
#  define SRT_HAS_X32 0
```

x86_64 supports the x32 ABI (0x40000000 < nr), so the BPF filter has an extra jump at the front to allow any x32 syscall (they all live in a tight range the BPF would otherwise have to handle). aarch64 omits that jump.

### Why static

The static binary has zero runtime dependencies on glibc/musl. It's the same size on all distros. We trust the kernel ABI more than we trust libc.

## 10.2 `vendor/srt-win-src/` (Rust)

20 kLOC Rust source. Cargo workspace (single crate).

### CLI Subcommands

```
srt-win install [--sandbox-user NAME] [--sublayer-guid GUID] [--proxy-port-range lo-hi] [--force]
srt-win uninstall [--sublayer-guid GUID]
srt-win exec -- <command>
srt-win wfp status
srt-win wfp verify
srt-win acl grant --sandbox-user-sid <sid> <path...>
srt-win acl revoke --sandbox-user-sid <sid>
```

When invoked as argv\[1] = `--srt-win` (multicall mode), the dispatcher routes to the same handlers.

### State DB

```
%LOCALAPPDATA%\sandbox-runtime\state.db
    TABLE sandbox_user
        username        TEXT
        password_dpapi  BLOB     ← CryptProtectData(machine scope)
        marker_version  INT
        sandbox_user_sid TEXT
        sandbox_group_sid TEXT
        created_at_unix  INT
    TABLE user_to_path         ← refcount of grant ACEs by path
        path TEXT
        holder_pid INT
        ACE VARBINARY
    TABLE deny_paths
        path TEXT
        holder_pid INT
        ACE VARBINARY
    TABLE ca_cert
        cert_der BLOB
```

The DB has a DACL of its own — only the real user (and administrators) can
open it; the sandbox user explicitly DENY'd. This is the only file-level gate on the credential.

### DPAPI Module

```rust
pub fn protect_machine(plaintext: &[u8]) -> Result<Vec<u8>> {
    let mut blob = CRYPT_INTEGER_BLOB::default();
    CryptProtectData(&CRYPT_DATA_BLOB{ cbData: plaintext.len() as u32, pbData: plaintext.as_ptr() },
                     None, None, None, None,
                     CRYPTPROTECT_LOCAL_MACHINE,
                     &mut blob)?;
    Ok(slice::from_raw_parts(blob.pbData, blob.cbData as usize).to_vec())
}

pub fn unprotect_machine(blob: &[u8]) -> Result<Vec<u8>> {
    let mut pb = CRYPT_INTEGER_BLOB::default();
    CryptUnprotectData(&CRYPT_DATA_BLOB{ … }, None, None, None, …, &mut pb)?;
    Ok(…)
}
```

Decryption requires the calling process to be `LocalSystem`, `Administrator`, or the **original user** (machine-scope flags to `CryptProtectData`/`CryptUnprotectData`). The broker runs with broker authority → can decrypt → passes plaintext to runner → runner turns the plaintext into a restricted token → child has the token but never the password.

### Two-Hop Launch

```
broker (real user, maybe non-admin)
    SpawnViaUACelevated(srt-win exec --pw <plaintext>)
        runner (NT-process, srt-sandbox user)
            ApplyRestrictedToken()
            ApplyJobObject()
            ApplyConsoleWindowStation()   ← so the child can write to a console if you want
            CreateProcess(...)
                child (srt-sandbox, restricted token)
```

The runner immediately zeroes the plaintext password after the token/restricted-logon is set up:

```
<SandboxCred as Drop>:
    for byte in self.pw.bytes_mut() { *byte = 0 }
    self.pw.clear()
```

### WFP Filter Installation

```rust
fn install() -> Result<()> {
    let engine = FwpmEngineOpen0(None, RPC_C_AUTHN_DEFAULT, None, None, None)?;
    FwpmTransactionBegin0(&engine, 0)?;
    FwpmSubLayerAdd0(&engine, &sublayer)?;
    for layer in [FWPM_LAYER_ALE_AUTH_CONNECT_V4, FWPM_LAYER_ALE_AUTH_CONNECT_V6] {
        FwpmFilterAdd0(&engine, &permit_loopback_filter_for(layer))?;
        FwpmFilterAdd0(&engine, &block_sandbox_user_filter_for(layer))?;
    }
    FwpmTransactionCommit0(&engine)?;
    Ok(())
}
```

Filter creation uses the `windows` crate's bindings. Each filter's conditions:

```rust
fn permit_loopback_conditions(port_range: (u16, u16)) -> Vec<FWPM_FILTER_CONDITION0> {
    vec![
        FWPM_FILTER_CONDITION0 {
            fieldKey: FWPM_CONDITION_IP_REMOTE_ADDRESS,
            matchType: FWP_MATCH_EQUAL,
            conditionValue: FWP_CONDITION_VALUE0 {
                value: FWP_CONDITION_VALUE0_0 {
                    v4_addr_and_mask: FWP_V4_ADDR_AND_MASK {
                        addr: 0x7F000000,    // 127.0.0.0
                        mask: 0xFF000000,    // /8
                    }
                }
            }
        },
        FWPM_FILTER_CONDITION0 {
            fieldKey: FWPM_CONDITION_IP_REMOTE_PORT,
            matchType: FWP_MATCH_RANGE,
            conditionValue: FWP_CONDITION_VALUE0 {
                value: FWP_CONDITION_VALUE0_0 {
                    range: FWP_RANGE0 {
                        valueLow: FWP_VALUE0 { uint16: port_range.0, … },
                        valueHigh: FWP_VALUE0 { uint16: port_range.1, … },
                    }
                }
            }
        }
    ]
}
```

The blocker condition is simpler:

```rust
FWPM_FILTER_CONDITION0 {
    fieldKey: FWPM_CONDITION_ALE_USER_ID,
    matchType: FWP_MATCH_EQUAL,
    conditionValue: FWP_CONDITION_VALUE0 {
        value: FWP_CONDITION_VALUE0_0 {
            sid: sandbox_user_sid,   // FWP_SECURITY_DESCRIPTOR_TYPE
        }
    }
}
```

### ACL Module

Uses Win32 `SetSecurityInfo()` / `GetSecurityInfo()`.

```rust
pub fn stamp_deny(sid: &str, paths: &[String]) -> Result<()> {
    let trustee = Trustee::from_sid(sid)?;
    for path in paths {
        let sd = get_security_info(path)?;
        let new_deny = explicit_deny_ace(&trustee, MODIFY_ACCESS)?;   // mask
        let new_deny_child = explicit_deny_ace(&trustee, FILE_DELETE_CHILD)?;
        let mut sd' = sd.clone();
        add_explicit_ace(&mut sd', &new_deny)?;
        if let Some(parent) = parent_dir(path) {
            let sd_parent = get_security_info(parent)?;
            let mut parent' = sd_parent.clone();
            add_inherited_ace(&mut parent', &new_deny_child)?;
            set_security_info(parent, &parent')?;
        }
        // store original for restore
        db.record_deny(path, current_pid, &sd)?;
        set_security_info(path, &sd')?;
    }
}
```

### CA Install

```rust
pub fn trust_ca(cert_der: &[u8]) -> Result<()> {
    let creds = read_setup()?.sandbox_creds;     // SandboxCred { user, pw }
    let token = LogonUser(&creds.user, …, &creds.pw, …)?;
    ImpersonateLoggedOnUser(token)?;

    let store = CertOpenStore(CERT_STORE_PROV_REG, …, 0, CERT_SYSTEM_STORE_CURRENT_USER, "Root")?;
    let der = CertCreateCertificateContext(X509_ASN_ENCODING, cert_der)?;
    CertAddCertificateContextToStore(store, der, CERT_STORE_ADD_REPLACE_EXISTING, nullptr)?;

    RevertToSelf()?;
    Ok(())
}
```

The token is dropped on every exit path; the registry write is gone.

## 10.4 Build Pipeline

```bash
# Linux (in CI):
gcc -static -O2 apply-seccomp.c -o dist-aux/seccomp-x64/apply-seccomp

# Cross-compile (CI does this for arm64):
aarch64-linux-gnu-gcc -static -O2 apply-seccomp.c -o dist-aux/seccomp-arm64/apply-seccomp

# Windows (in CI):
cargo build --release --target x86_64-pc-windows-msvc
cargo build --release --target aarch64-pc-windows-msvc

# Universal TypeScript:
tsc           # → dist/
```

`npm run prepare` runs `husky`, `npm run prepublishOnly` runs `clean && build`. CI runs `build:seccomp` before `test` on Linux arches only.

## 10.5 Multicall Binary Dispatch

Some embedders want to **fold** `srt-win` into their own larger binary
(an installer, for example). The orchestrator supports this via
`SRT_WIN_DISPATCH_ARG1 = '--srt-win'`. When `windows.srtWin.path` is set
to the embedder's binary, every invocation prefixes `--srt-win` as
argv\[1] so the embedder's dispatcher can route to `srt_win::run_from_args`.

```
my-embedder.exe ...
my-embedder.exe --srt-win exec -- ...
```

Why argv\[1] and not argv\[0]? Windows `CreateProcessWithLogonW` and `ShellExecuteExW(runas)` don't preserve a spoofed argv\[0]. argv\[1] is part of the literal command line and is preserved.

## 10.6 Why Two Architectures and Not Three?

x86_64 and aarch64 are the supported targets. There is no serious
production deployment on 32-bit. i386/armhf is **not** supported, and
`checkLinuxDependencies` returns an error rather than silently degrading.

The seccomp filter is architecture-specific (different syscall numbers,
different arg layouts); it has to be compiled per-arch. The Rust helper
isn't architecture-specific in the syscall sense but is built per-arch
because Rust targets differ.

## 10.7 What the Orchestrator Never Touches

- It does not call `NtCreateProcess` directly — that's the runner's job.
- It does not write to `HKLM` — that's the installer's job (elevated).
- It does not call `CreateToolhelp32Snapshot` — the violation monitor uses `/proc/<pid>/{cwd,fd/N}` only.
- It does not need `windows-rs` Win32 headers — it imports the wrapper as a Node module that proxies JSON over stdio to the `srt-win` binary. The orchestration surface is small (just the argv).

This is **why** there's a Rust helper at all: to keep the orchestrator's Win32 surface area to zero. The orchestrator stays portable Node.
